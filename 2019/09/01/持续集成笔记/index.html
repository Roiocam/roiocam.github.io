持续集成笔记

Posted by Andy Chen on 2019-09-01

持续集成在软件开发协作中是一个重要阶段。

持续集成的“典型”流程

持续集成（CI）是软件开发周期中的一个阶段，来自不同团队成员或不同功能的代码集成在一起。这通常涉及在短暂的环境中合并代码（集成），构建应用程序和执行基本测试。

过去，代码集成在软件开发生命周期的“集成阶段”。这个阶段是在不同的团队花费数周，数月甚至数年独立工作，致力于不同（部分）应用程序或服务之后。你可以猜到项目的整合阶段会有多痛苦。在这个阶段花费数周甚至数月并不罕见。这是在瀑布时代。

在极限编程（XP）和敏捷编程中，集成变得频繁，开发人员通常在单元完成后尽快集成。这是在共享源代码存储库上完成的。频繁的集成变得自动化且连续，这促使在集成新代码之前需要进行某种检查。因此，持续整合。

CI工作流程因工具，编程语言，项目和许多其他因素而有很大差异，但常见的流程包含这些步骤。

- 推送代码到代码存储库（Git)
- 静态分析
- 部署前测试
- 打包和部署到测试环境
- 部署后测试

下面来看看这个流如何在Java项目中起作用。

一.代码仓库

通常，会有一个代码存储库和某种用于提供新代码的工作流。根据工作流程，提交代码将启动CI管道，这通常从静态代码分析开始。您可以参考 这篇文章 来帮助选择合适的工作流程。

二.静态分析

静态（代码）分析在应用程序的代码库上完成，无需运行软件。这里的目标是确保代码没有可能的错误并符合标准格式和样式。

让我们添加 FindBugs 来检查可能的错误和 Checkstyle 以确保项目符合编码标准（我们将使用 Google Java Style）。

在此阶段，任何测试都可以在不部署到服务器的情况下运行。这将包括单元测试和各种其他类型的测试（可能是功能或集成）。 此阶段用于确保更改集不会破坏功能并与其他部分代码一起使用，因为测试是在整个代码库上运行的，而不仅仅是新的更改（正如作者可能在dev env上所做的那样） 。

三.打包并部署到测试/暂存环境

根据项目的类型，应用程序可以构建，打包，发送到测试或登台环境（模仿生产）。这可确保集成更改与其他部件良好地构建，并且可以部署以进行功能测试。正是在这个阶段，我们还验证了新的更改是否与其他库和部署环境兼容。这个阶段也应该是自动化的。

对于需要部署应用程序的测试，CI管道的此阶段是它们运行的​​位置。测试根据应用程序的工具，框架和语言而有所不同，但它们通常是功能集成和性能测试。该阶段的成功执行结束了变更集的CI管道，表明它对用户来说足够好。

CI管道成功完成后，部署的应用程序可以由“用户”或QA团队进行手动测试，以确保其符合客户的要求。现在可以将CI管道生成的包或工件用于生产。这也可以通过成功实施Continuous Delivery（CD）管道实现自动化。
