MongoDB的副本集群和分片集群

Posted by Andy Chen on 2019-06-25

简介：在 MySQL ，我们对数据进行备份时的通用做法是主从复制，但是主从复制的本质还是一个单福本应用。在 MongoDB 中，推荐建立副本集保证服务的容错性。

一、副本集

概念

副本集在多台机器同步数据，提供了数据冗余，扩展了数据可用性。避免服务在故障时导致丢失数据。除此之外在一些场景还有扩展读性能的作用。

副本集和主从同步相似，主MongoDB实例接受所有的写操作， 其他实例接受主 MongoDB 数据保证数据同步。

结构原理

副本集不同于以往的主从模式。

在集群主MongoDB实例故障时，副本集自动投票，选举出新的Master，并引导其余的Slave服务器联机新的Master，可以说MongoDB是自带故障转移的主从复制。

心跳检测：整个集群需要保持一定的通信才能知道哪些节点活着哪些节点挂了，MongoDB节点每2秒向其他节点发送一次Ping包以检测节点存活。

数据同步：副本集分为初始化同步和Keep复制。初始化同步指全量从主节点同步数据，如果主节点数据量大则同步时间长。Keep复制则一般为增量同步。

副本集选举过程和注意点

MongoDB副本集选举采用的是Bully算法，这是一种协调者（主节点）竞选算法，主要思想是集群的每个成员都可以声明它是主节点并通知其他节点。

选举过程：

  1. 得到每个服务器节点最后的操作时间
  2. 如果集群大部分的机器宕机，保留活着的节点都为secondary并停止，不选举。
  3. 如果集群选举出来的主节点或者所有从节点最后一次同步时间很旧，则停止选举等待手动操作。
  4. 如果都没问题则选择最后操作时间戳最新的服务器。

特点：参与选举的节点数量必须大于副本集总节点数量的一半。

MongoDB同步延迟

即修改过的信息在集群中不改变，删除的数据仍在。

首先出现同步延迟的地方必然就有数据同步，在MongoDB中有两种数据冗余方式，一种是主从，一种是副本集。MongoDB在执行操作时会记录写日志（oplog），由于其他副本更新oplog的延迟，则会导致同步延迟

二、分片集群

分片集群是一种可以水平扩展的模式，在数据量很大时很给力，大规模应用一般会采用这种方式，分片集群有效的解决了单台服务器硬件资源限制，将数据水平拆分出去。降低单点压力，

概念：

将数据集合切分为一个个小块，分散到不同的服务器中。

构建MongoDB分片集群需要三个角色：

  1. 分片服务器：用来存储实际的数据集
  2. 配置服务器：独立的MongoDB进程，保存集群和分片的元数据，即各分片包含了哪些数据信息。
  3. 路由服务器：前端路由，客户端由此接入，让整个集群看起来像单一的数据库。
