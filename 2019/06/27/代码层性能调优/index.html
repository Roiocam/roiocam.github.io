代码层面性能调优

Posted by Andy Chen on 2019-06-27

介入调优的时期

代码编码时

保证有效编码。

  1. 减少磁盘I/O操作
  2. 降低竞争锁
  3. 使用高效算法
  4. 设计价格这种包含折扣、红包活动时，使用装饰模式设计这个业务。

    性能测试时

分析预期数据和压力，进行压力测试，通过性能分析，统计工具来统计各项性能指标

监控性能

分析日志监控以及性能统计日志。

哪些数据可以体验性能

CPU

  1. 代码递归导致无限循环
  2. 正则表达式引起回溯（）
  3. JVM频繁Full GC
  4. 多线程造成大量上下文切换

内存

读写性能一般无瓶颈，但是内存成本高，空间有限，注意内存空间占满后无法回收导致的内存溢出和内存泄露

磁盘

主要体现在I/O读写性能瓶颈

网络

主要体现在带宽瓶颈

异常

抛出异常时需要构建异常栈，对异常捕获和处理，这个过程非常耗费系统性能，如果在高并发下引发异常，持续的处理异常，则会导致影响系统性能。

数据库

数据库一般为磁盘I/O的读写，大量的数据库操作等同于大量的磁盘I/O，会导致磁盘I/O性能瓶颈，影响数据库操作延迟。

锁竞争

在多线程中，为了保证数据原子性（共享资源被一个线程独占，其他线程无法修改）时就会用到锁。锁的使用会带来上下文切换。JDK1.6之后JAVA对JVM内部锁做了多次优化，新增了偏向锁，自旋锁，轻量级锁，锁粗化，锁消除。

指标

1.响应时间

  1. 数据库响应时间：一般为整个请求链最耗时
  2. 服务端响应时间：Nginx分发请求所消耗的时间和服务端执行的时间
  3. 网络响应时间：网络传输时，网络硬件需要对传输请求解析的耗时。
  4. 客户端响应时间：客户端渲染数据等耗费的时间

2.吞吐量

在测试时，往往比较注重系统接口的TPS(每秒事务处理量)。在系统中可以把吞吐量分为磁盘吞吐量和网络吞吐量。

磁盘吞吐量的关键衡量指标

  1. IOPS：每秒输入/输出量（读写次数），关注的是随机读写性能。适应于随机读写频繁应用(小文件存储[图片]，OLTP数据库，邮件服务器)
  2. 数据吞吐量：单位时间能够成功传输的数据量，对大量顺序读写频繁的应用，（视屏编辑，视频点播），传输大量数据时，数据吞吐量是关键衡量指标

网络吞吐量的关键衡量指标

设备能接受的最大数据传输速率。

3. 计算机资源分配使用率

  1. CPU占用率
  2. 内存使用率
  3. 磁盘I/O
  4. 网络I/O

4. 负载承受能力

观察系统上升时间的上升曲线是否平缓，如果出现明显断层（抛出大量错误或者抛出大量异常），则证明系统处理到了极限。

延伸1：CPU利用率和系统负载两个指标的关系。

系统负载：单位时间内正在运行或者等待的进程数或者线程数，代表系统繁忙度

CPU利用率：单位时间一个线程或进程实时占用CPU的百分比。因为一个线程运行时未必都在实时的利用CPU。

在CPU密集型应用中，系统负载未必高，但利用率肯定高，一个线程一直在计算。

在I/O密集型应用中，可能CPU利用率不高，但系统负载非常高，因为I/O经常阻塞，导致很多线程/进程处理阻塞等待状态。

延伸2：如何避免业务异常生成栈追踪信息

JVM在异常抛出时，会构建栈追踪信息，整个过程十分耗费性能。一般业务异常避免生成堆栈追踪信息，我们知道这个异常是什么原因，所以直接返回字符串就好了。而系统异常，一般都会生成堆栈追踪信息，以便追踪源头，更好的排查问题。通过自己实现自定义异常，继承RuntimeException，然后设置writableStackTrace为false可避免在异常时构建异常栈。
