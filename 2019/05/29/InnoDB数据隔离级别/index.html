数据库 事务

InnoDB数据库隔离级别

Posted by Andy Chen on 2019-05-29

一次封锁

在大量并发访问的应用里，为了防止死锁，一般推荐使用一次封锁法：在方法的开始阶段，已经预先知道会用到哪些数据，全部锁住，在运行之后全部解锁。

这种方法可以有效避免循环死锁。却不适用于数据库中。

因为在事务开始阶段，数据库并不知道会用到哪些数据。

二段锁

数据库遵循的是二段锁协议，将事务分为两个阶段，加锁和解锁阶段。

  • 加锁阶段：在该阶段可以进行加锁操作，在对任何数据进行读操作之前要申请获取S锁（共享锁，其他事务可以继续加共享锁，但不能加排它锁），在写操作之前要申请获得X锁（排它锁，其它事务不能再获取任何锁）。加锁不成功，则事务进行等待状态。（阻塞）。
  • 解锁阶段：当事务释放一个锁以后，事务进入解锁阶段，在该阶段只能进行解锁操作不能再进行加锁操作。

二段锁协议无法避免死锁，但是二段锁协议可以保证事务的并发调度是串行化（串行化很重要，尤其是在数据恢复和备份时）

事务加锁方式

事务的四种隔离级别

隔离级别 脏读 不可重复读
未提交读 可能 可能
已提交读 不可能 可能
可重复读 不可能 不可能
可串行化 不可能 不可能
  1. 未提交读：允许脏读，也就是可能读到其他会话中未提交事务修改的数据
  2. 提交读：只能读取到已经提交的数据。Oracle等多数数据库默认都是该级别
  3. 可重复读：在同一个事务内查询的事务都是开始时刻一致的,InnoDB默认级别，在SQL标准中，该隔离级别消除了不可重复读，但是还是存在幻想读。
  4. 串行读：完全串行化的读，每次读都需要获取表级别的锁。读写相互都会阻塞。

未提交读一般数据库不会使用。

MySQL中锁的种类

常见表锁、行锁，也有新加入的Metadata Lock等。

  • 表锁是对一整张表加锁，虽然可分为读锁和写锁，但毕竟是锁住了整个表，会导致并发性能降低，一般是做DDL处理时使用。
  • 行锁是锁住数据行，这种加锁方法比较复杂，但是由于只锁住有限的数据，对其他数据没有限制，所以并发性能强。MySQL一般用行锁来处理并发事务。

已提交读

由于MySQL的InnoDB默认是使用的RR（可重复读）级别，所以我们先要将session开启成RC（可提交读），并设置binlog的模式。

事务A 事务B
begin; begin;
update class_teacher set class_name’初三三班’ where teacher_id=1 update class_teacher set class_name’初三三班’ where teacher_id=1
commit;

为了防止并发过程的修改冲突，事务A中MySQL给teacher_id的数据行加锁，并一直不commit;那么事务B就一直拿不到行锁，wait超时。

这里需要注意的是，teacher_id是有索引的，如果where指定的是没有索引的class_name.则update class_teacher set teacher_id =3 where class_name = ‘初三三班’ 则会对所有的数据行加锁。因为MySQL不知道哪些数据行是class_name =’初三三班’，如果一个条件一直无法通过索引快速过滤，存储引擎层面就会将所有记录加锁后返回，再由MySQL Sever层进行过滤。

但在实际使用时，MySQL做了一些改进，在MySQL Sever过滤条件时，发现不满足后会调用unlock_row,把不满足的数据行释放锁。（违背了二段锁协议的约束）保证了最后只有满足条件的数据行持有锁。即使是MySQL也会为了效率违背规范。

这种情况同样适用于MySQL的默认隔离级别RR。当做一个数据量很大的表做批量修改时，如果无法使用相应索引。MySQL Server过滤数据的的时候特别慢，就会出现虽然没有修改某些行的数据，但是它们还是被锁住了的现象。

可重读

这是MySQL中InnoDB默认的隔离级别。这里分为“读”和“写”两个模块。

读

读就是可重读，一事务的多个实例在并发读取数据时，会看到同样的数据行。

RC(不可重读)模式下的展现

事务A 事务B
begin; begin;
select id,class_name,teacher_id from class_teacher where teacher_id = 1 .\ \ \ \ [id=1 class_name=’初三二班’ teacher_id = 1 , id=2 class_name=’初三一班’ teacher_id=1]
update class_teacher set class_name ‘初三三班’ where id =1
commit;
select id,class_name,teacher_id from class_teacher where teacher_id = 1 .\ \ \ \ [id=1 class_name=’初三三班‘ teacher_id = 1 , id=2 class_name=’初三一班’ teacher_id=1] 这里读到了事务B修改的事务，和第一次查询到的结果不一致，是不可重读
commit;

再来看看RR级别中MySQL的表现：

事务A 事务B
begin; begin; begin;
select id,class_name,teacher_id from class_teacher where teacher_id = 1 .\ \ \ \ [id=1 class_name=’初三二班’ teacher_id = 1 , id=2 class_name=’初三一班’ teacher_id=1]
update class_teacher set class_name = ‘初三三班’ where id =1 ;
commit;
insert into class_teacher values (null,’初三三班’,1);
commit;
select id,class_name,teacher_id from class_teacher where teacher_id = 1 .\ \ \ \ [id=1 class_name=’初三二班’ teacher_id = 1 , id=2 class_name=’初三一班’ teacher_id=1] 没有读取到事务A修改的数据，和第一次sql读取的一致，是可重复读。也没有读取到事务C添加的数据。
commit

不可重复读和幻读的区别

这两者有些相似，经常容易搞混。但是不可重复读的重点在于update和delete，而幻读的重点在于insert。

如果使用锁机制来是实现这两种隔离级别，在可重复读中该SQL第一次读取到数据后就对这些数据加锁。其他事务则无法修改这些数据。（这种方法无法锁住insert的数据）此时当事务A读取了数据或修改了全部数据，事务B仍然可以使用Insert提交数据，这时在事务A中就会莫名其妙多出了新增数据，这就是幻读，无法通过行锁避免。

如果需要避免幻读，则需要Serializable隔离级别，读取用读锁，写用写锁，读锁写锁互斥（任意时刻只能有一个线程访问该对象，无论是写还是读）。这种方式可以避免幻读，不可重复读，脏读等问题。但会极大降低数据库的并发性能。

不可重复读和幻读的最大区别，就在于如何通过锁机制解决他们产生的问题。

悲观锁和乐观锁

悲观锁

正如其名，悲观锁指的是对数据被外界修改持保守态度。在整个数据处理过程中，数据将处于锁定状态。悲观锁的实现依靠数据库提供的锁机制（也只有数据库提供的锁机制能够真正保证数据访问的排他性）

在悲观锁的情况下为了保证事务的隔离性，需要一致性锁定读，读取数据时加锁，其他事务无法修改这些数据，修改时也要加锁，其他事务无法读取这些数据。

乐观锁（MVCC-多版本并发控制）

相对于悲观锁，乐观锁采取了更为宽松的加锁机制，悲观锁依靠数据库的锁机制保证操作最大程度上的独占性。但随之带来的是数据库性能的大量开销。

乐观锁在一定程度上解决了这个问题。乐观锁大多数是基于数据版本（version）记录机制实现。即为数据增加一个版本标识，读取数据时一同读取版本号，更新时则对版本号自增一个version. 此时将提交数据的版本和数据库表中对应记录的当前版本进行对比。如果提交的数据版本大于数据库当前版本才给予更新，否则是过期数据。

MVCC的实现并没有规范，每个数据库都会有不同的实现方式。

MySQl InnoDB的MVCC实现

在InnoDB中，会在每行数据后添加两个额外隐藏值来实现MVCC，这两个值一个记录这行数据何时被创建，另外一个记录这行记录何时过期（或者被删除）。在实际操作中，存储的并不是时间而是事务的版本号。每开启一个事务，事务版本号就会递增。在可重读事务隔离级别下：

  • SELECT时，读取创建版本号<=当前事务版本号，删除版本号为空或>当前事务版本号。
  • INSERT时，保存当前事务版本号为行的创建版本号。
  • DELETE时，保存当前事务版本号为行的删除版本号
  • UPDATE时，插入一条新纪录，保存当前事务版本号为行创建版本号，同时保存当前事务版本号到原来删除的行

通过MVCC，虽然每行记录都增加了额外的存储空间、更多的行检查工作以及额外的维护工作，但减少了锁的使用，大多数读操作都无需加锁。读操作简单，性能好，并且保证只会读到符合标准的行，也只锁住了必要行。

在网络和教科书上，大多都是和上文事务的四种隔离级别这一模块列出的意思。

RR级别是可重复读的，但是无法解决幻读，只有在Serializable级别才能解决幻读。但是在实际的MySQL中不存在这种情况，可见MySQL的RR级别是解决了幻读的读问题的。

读问题解决了，根据MVCC定义，并发提交数据时会出现冲突，那么冲突如何解决？再看看InnoDB中RR级别对于写数据的处理。

“读”与“读”的区别

事务的隔离级别其实都是对于读数据的定义，但在这里被拆为了读和写两个模块来讲解。这主要是因为MySQL中的读和事务隔离级别中的读不一样。

在RR级别中，通过MVCC机制，虽然数据变成可重复读，但是数据是历史数据不是实时数据，对于一些数据时效性很敏感的业务就可能出现问题。

这种读取历史数据的方式称为快照读（snapshot read)，而读取数据库当前版本数据的方法叫做当前读（current read)。显然在MVCC中：

  • 快照读：就是select （select * from table….）
  • 当前读：特殊的读操作，插入/更新/删除操作，属于当前读，处理的都是当前数据，需要加锁。
    • select * from table where ? lock in share mode;
    • select * from table where ? for update;
    • insert
    • upadte
    • delete;

事务的隔离级别实际上都是定义了当前读的级别，MySQL为了减少锁处理（包括等待其他锁）的时间，提升并发能力，引入了快照读的概念，使得select无需加锁，而update、insert这些“当前读”，就需要另外的模块来解决。

写（“当前读”）

为了解决当前读中的幻读问题，MySQL事务使用了Next-Key锁。

Next-Key锁是行锁和GAP（间隙）锁的合并，行锁上文已经介绍了，接下来说GAP（间隙锁）。

行锁可以防止 不同事务版本的数据修改提交时造成数据冲突的情况。但是如何避免别的事务插入数据成了问题。

RC级别事务：

事务A 事务B
select * from class_teacher where teacher_id=30;\ \ {id=2,class_name=’初三二班’,teacher_id=30}
update class_teacher set class_name=’初三四 班’ where teacher_id=30;
insert into class_teacher value(null,’初三二班’,30); commit;
select * from class_teacher where teacher_id=30 \ \ \ \ [{id=2,class_name=’初三四班’,30},{id=10,class_name=’初三二班’,teacher_id=30}]

RR级别：

事务A 事务B
select * from class_teacher where teacher_id=30;\ \ {id=2,class_name=’初三二班’,teacher_id=30}
update class_teacher set class_name=’初三四 班’ where teacher_id=30;
insert into class_teacher values (null,’初三二班’,30); 阻塞等待中….
select * from class_teacher where teacher_id=30;\ \ {id=2,class_name=’初三四班’,teacher_id=30}
commit;
事务Acommit后，事务B的Insert才开始执行。

在RC级别中，事务A修改teacher_id=30的数据，但是当事务Binsert插入新数据后，事务A莫名其妙多了一行teacher_30的数据，而且没有被之前update语句所修改，这就是“当前读”的幻读。

RR级别中，事务A在update后加锁，事务B无法插入新数据，这样事务A在update前后读的数据保持一致，避免了幻读，这个锁就是GAP锁。

MySQL的实现：在class_teacher表中,teacher_id是个索引，它维护了一套B+树的数据关系，InnoDB使用的是聚集索引，teacher_id是二级索引，需要维护一个索引字段和主键id的树状结构，保证顺序排列。InnoDB将这段数据分成了几个区间：

  • (-无穷,5]
  • (5,30]
  • (30,+无穷)

update class_teacher set class_name=‘初三四班’ where teacher_id=30;不仅用行锁，锁住了相应的数据行；同时也在两边的区间，（5,30]和（30，+无穷 ），都加入了gap锁。这样事务B就无法在这个两个区间insert进新数据。

update的teacher_id=20是在(5，30]区间，即使没有修改任何数据，Innodb也会在这个区间加gap锁，而其它区间不会影响，事务C正常插入。

受限于这种实现方式，Innodb很多时候会锁住不需要锁的区间。

如果使用的是没有索引的字段，比如update class_teacher set teacher_id=7 where class_name=‘初三八班（即使没有匹配到任何数据）’,那么会给全表加入gap锁。同时，它不能像上文中行锁一样经过MySQL Server过滤自动解除不满足条件的锁，因为没有索引，则这些字段也就没有排序，也就没有区间。除非该事务提交，否则其它事务无法插入任何数据。

行锁防止别的事务修改或删除，GAP锁防止别的事务新增，行锁和GAP锁结合形成的的Next-Key锁共同解决了RR级别在写数据时的幻读问题

Serializable

这个级别很简单，读取加共享锁，写加排它锁，读写互斥。使用的悲观锁理论，实现简单，数据更加安全，但是并发能力非常差。如果业务并发特别少，又要求数据及时且可靠则可以使用这种模式。另外在Serializable这个级别，select也会加锁。
