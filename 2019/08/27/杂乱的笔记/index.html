杂乱的笔记

Posted by Andy Chen on 2019-08-27

Linux

  • 一切都是文件。在Linux中，诸如键盘，显示器等设备，Socket等都被设计为文件。这样的好处是对于程序员来说，一切的操作都将原子化（读，写）
  • 五种I/O模型(阻塞IO，非阻塞IO，IO多路复用，信号驱动IO，异步IO)。在Linux Network I/O中，涉及两个系统对象（调用程序和系统内核）
    • 阻塞I/O（读操作：内核开始准备数据，阻塞等待数据；用户进程阻塞，等待内核数据。实际上两个阶段都阻塞）
    • 非阻塞I/O（读操作：内核准备数据，阻塞等待；用户进程不阻塞，论询内核数据是否准备好。实际上只有内核阻塞）
    • I/O多路复用（读操作：Socket I/O设置为非阻塞。用户进程调用select阻塞，等待多个Socket的数据。性能差于阻塞I/O，因为了内核进行了两次系统调用，但是多路复用可以监听多个Socket,能够处理更多的连接）
    • 异步I/O（读操作：用户不阻塞，内核不阻塞立即返回，当内核数据准备完成后，写入用户内存并发送信号通知用户进程）
    • 非阻塞I/O在内核未准备好数据时不阻塞，但是在内核调用recvfrom将数据拷贝到内存中时，阻塞进程。

Java

性能测试，基准测试

  • 利用JMH进行基准测试
    • 避免死码消除（JIT会自动优化那些无用代码，如执行后没有被引用的对象，将会被JMH忽略，以至于测试时相当于没有执行）
  • 并发锁
    • 单个程序内，无外部操作。一般基于JVM的锁解决并发问题。
    • 需要操作数据库。基于数据库行锁解决并发问题，不同数据库引擎性能不同。
    • 分布式场景下：分布式锁（可以用MySQL，Redis，Zookeeper做监视器（monitor),分别对应三种共享组件：数据库，缓存，分布式协调组件。

      持续集成，持续部署，持续交付

  • 进行Junit测试时，充分覆盖。
  • 每个业务都需要编写测试代码。（CI/CD）

    线程Future

  • 将来式（阻塞）：提前编写call()，使用future.get();阻塞主线程，在任务完成后返回结果。
  • 将来式（非阻塞）：提前编写call()，论询isDone()函数，当返回true时调用future.get();返回结果。
  • 回调式(Netty，Guava，CompletableFuture)：提前编写call()，为future.addListener()添加监听，在任务完成时触发钩子执行call()。主线程不会被阻塞。
  • JDK1.8中CompletableFuture的thenCompose*（）方法可以实现回调的回调。减少回调地域。

    并发计数器

  • AtomicLong：并发下读性能优秀，写性能急剧下降，内存需求低，不适合作为高性能的计数器。
  • LongAddr：并发下写性能优秀，读性能因为组合求值比较差，占用内存因为Cells数组和缓存行填充所以占用大。
  • ConcurrentAutoTable：和LongAddr接近的写入性能，更差的读性能。

网络

长链接相关

  • 长连接的保活是难点。
  • KeepAlive在应用层开启，但是KeepAlive的参数（重试间隔时间等）在操作系统层面设置（在Linux下为/etc/sysctl.conf）
  • 长连接用于对性能有要求的框架（如RPC框架）
  • 长连接需要设计定时任务，定时发送心跳包保活。除了定时任务，还需要在协议层面支持心跳包（防止将心跳请求为正常流量，干扰服务器正常业务）
