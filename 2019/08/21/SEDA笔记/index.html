I/O 并发 线程模型

SEDA笔记

Posted by Andy Chen on 2019-08-21

简介

SEDA是分阶段事件驱动架构的首字母缩写，它将复杂的事件驱动应用程序分解为阶段网络，阶段之间依靠队列连接，应用程序设计者只负责每个阶段的服务逻辑和阶段之间的连接逻辑，而由各个阶段自身负责资源的管理和负载的适应功能。此设计避免了与基于线程的并发模型相关的高开销，并将事件和线程调度与应用逻辑分离，使系统对负载变化的良好适应性以及良好的系统缩放性，达到高并发，高效率。

通过对每个事件队列执行准入控制，可以很好地加载服务以加载，从而防止当需求超过服务容量时资源过度使用。 SEDA采用动态控制来自动调整运行时参数（例如每个阶段的调度参数），以及管理负载，例如，通过执行自适应减载。 将服务分解为一组阶段还可以实现模块化和代码重用，以及为复杂事件驱动的应用程序开发调试工具。

值得关注的问题

  1. 如何合理地进行资源初始化分配：如果未能进行合理地线程以及队列资源分配，导致出现资源需求较大的阶段被分配了数量不足的线程数，或者资源需求较小的 阶段被分配了数量过多的线程数，则会造成系统吞吐量的下滑，甚至会导致系统性能在高并发下远低于多线程模型系统。

  2. 如何合理地对系统进行阶段划分：不合理的阶段划分将导致系统性能下滑。

    两种常用的互联网服务器模型

    1. 多线程编程模型

    最简单最普遍的服务器编程模型，如Servlet就是采用这种编程模式，如下图：

该模型针对每一个请求，都会创建分配一个线程，这个线程负责这个请求的处理。

优点：

执行粒度是一个完整的处理流程，处理逻辑清晰，容易开发。

缺点：

如果处理过程中某一步骤的处理时间过长（网络调用，磁盘I/O读写），则会阻塞后续请求，造成资源浪费。并且随着处理请求的不断增加，导致并发执行的线程数量太多。过多的线程数量导致系统在线程调度和资源抢占上的开销过大。

2. 事件驱动模型

事件驱动模型的思想是：将处理流程分割成多个步骤，每个步骤都实现为一个有限状态机（FSM）所有的处理请求会作为事件进入系统，由调度器负责传递给相应的状态机，
状态机的处理结果也以事件的形式传给调度器，新的时间再次被调度器转发给下一个状态机进行处理，直至处理完成。

优点：

将各个步骤独立实现，方便进行系统监测和调整。

缺点：

调度程序的设计和实现过于复杂，针对不同的应用和系统的逻辑变更需要不同的实现，导致采用这一模型构建出来的系统十分庞大和难以控制。

总结：

综上所述，主流的多线程编程模型和事件驱动模型虽然都各有优势，但同时也有不可避免的缺陷，并不是适合在高并发的成熟系统下作为编程模式。

阶段性时间驱动模型

针对上述主流并发架构的缺陷，汲取多线程和事件驱动模型的优点，最大程序规避两者缺点，SEDA应运而生。SEDA整合了多线程和时间驱动服务器模式的优势，它可以高效地管理和控制服务器资源，良好的适应高并发环境。

应用程序服务器之请求处理步骤通常是复杂的，基于事件驱动处理的有穷状态机（FSM）所以实现服务器程序过程就类似用程序实现有穷状态机过程。

SEDA架构能对有穷状态机进行分析，然后将相关关状态聚集在同一Stage中，Stage间采用队列的方式来进行通信。每一个Stage皆完全独立，均拥有自己的线程池，以及为了专门处理到达这一步骤所必须进行的工作。所有的Stage均通过自身事件队列连接在一起，构成完整的请求处理网络。性能控制器和动态线程池依请求的繁忙程度动态来调整线程池的大小，以达到系统资源的最有分配。每一个Stage由下述四部分组成：

  1. 事件队列：用以维持Stage间通信
  2. 事件处理器：用以执行请求到这一个Stage中所应执行的工作
  3. 线程池：用以提供事件处理器且可以并发执行事件之环境
  4. 性能控制器：用以对该Stage资源（线程数，队列长度等）进行调整

通过这四部分的协作配合，每一个Stage都可以很好地运行，并且可以控制资源的使用。已经过Stage处理完，若没有后续工作，即可以回收线程池中的线程，来供给其他Stage使用。Stage结构如下图所示。

在SEDA架构中，基本的处理单元称为阶段（Stage），一个阶段由事件队列、动态线程池、事件处理器和一个性能控制器四个组件构成。SEDA将一个请求的处理过程分解为一系列的阶段，阶段之间通过事件队列联系，开发人员只负责每个阶段的服务逻辑以及阶段间的连接逻辑，而由各个阶段自身负责资源管理以及负载适应功能。使用这种解耦拆分可以使系统达到高并发性、对负载变化的良好适应性以及高度的可缩放性。

自实现SEDA异步框架

(一). Stage结构

在这个框架中，一个Stage的组件有：

  1. D-MQ ：消息中间件。用作消息队列，传递消息。
  2. Local-Queue：本地队列，一般是BlockingQueue，用以辅助实现Stage内的动态线程池。采用LocalQueue的目的是避免数据在mq中堆积导致mq性能下降。
  3. ThreadPool：动态线程池，进行事件并发处理。
  4. Worker：事件的具体处理器
  5. StageController：State的性能控制器。用以对Stage的队列，资源，调度策略进行控制。

(二). SEDA框架使用场景

  1. 系统资源监控(CPU，内存，线程池，队列)
  2. 外围服务交互情况（API调用，上游服务交互，请求方）监控
  3. 系统报警（服务异常，接口压力过大）
  4. 基于日志和事件的数据挖掘（规则挖掘）
  5. 重要业务数据切片存储（里程碑消息，核心服务交互数据）
  6. 异步触发的操作（表A写完后异步写表B）

(四). 异步框架原生态结构(Virtual Bundle)

异步框架在无任何扩展时，其主要组件如下：

  1. bundle：消息中心的核心组件。由读，处理，写三部分组成。同时整合开关，定时器，动态线程池等元素来支持多样化的输入和需求。bundle可以从多种数据源获取数据，并进行数据处理。
    • 开关：用以决定该bundle是否被激活。如未被激活，则该bundle将停止读取数据，同时不会在其他服务上产生该bundle对应的数据（比如在mq上生成该bundle的队列、连接、交换机等。）
    • 定时器：用以指定该bundle是否定时运行。如未指定，则实时运行
    • 动态线程池：用以支持bundle以同步/异步方式调用。如未指定，则同步运行。
  2. Bundle Decider：用以对bundle的关键指标进行决策（是否激活，时效性，同步类型），并同时提供健康检查。
  3. Work Carrier：处理数据的最小单元。
