构建高性能应用服务器所需要知道的内容

Posted by Andy Chen on 2019-08-30

本文节选翻译自teskalabs的博客.作者为Ales Teska.

本文主要介绍了构建高性能应用服务器所涉及的一些模型架构等知识。

构建应用程序服务器并非易事。 它需要深入了解各种组件，如网络或OS内核。 我花了几年时间在软件工程的迷人领域：高性能应用服务器，如HTTP服务器，SSL终端，API网关，最新的软件定义网络（SDN）或网络功能虚拟化（NFV）工具。

这篇文章详细阐述了我在这次旅行中发现的主要发现。我谦卑地希望它们对从事这一令人兴奋的领域的人们有用。

反映堆模式

反应器模式由两个主要元素组成:事件循环和事件。事件由OS内核触发，以处理各种情况，例如传入的网络数据包。应用程序服务器在事件循环期间休眠，并被此类事件的到来唤醒。然后由应用程序代码处理事件，服务器返回到事件循环。

事件循环构建在著名的POSIX函数(如select()、poll()或它们的现代替代方法epoll())之上。另一个操作系统提供了各种方法来实现相同的效果，如BSD或macOS中的kqueue。

这种体系结构提供了出色的性能，原因有二:它提供UNIX设计匹配和非阻塞IO操作。应用程序不需要在IO非阻塞模式下等待读或写完成—它可以同时处理其他事件。这是一个如何提供更高的IO性能的秘诀。

反应器模式与UNIX设计相匹配，应用程序在很大程度上保持非活动状态，并且只对OS内核传播到用户空间的输入和输出事件做出响应，以便优化利用可用的系统资源。

入站和出站

常见的网络协议是双向的、全双工的，例如底层的TCP连接或上层的HTTP协议。TCP连接可以以半关闭状态存在，并且仍然以剩余的打开方向传输数据。

HTTP协议利用了这种半关闭状态。在HTTP请求中指示数据结束的一种方法是在请求数据发送到服务器后关闭连接的出站方向。然后从这个半关闭的连接接收HTTP响应，该连接最终由服务器关闭。半关闭状态是通过shutdown()系统调用来实现的。

为了处理这些情况，应用服务器应该尽可能独立地实现入站和出站方向的处理。对于响应用于接收相应请求的相同网络连接的应用程序服务器也是有效的。考虑HTTP服务器和HTTP代理之间的区别。

最重要的是，入站和出站也是应用程序事务的开始和结束。 应用程序服务器事务以读入事件开始，该事件由入站网络数据，read（）系统调用，应用程序逻辑启动，并在出站部分结束。

基于写入“就绪”，事务可以执行所谓的直接write()系统调用，或者将出站数据存储在出站缓冲区中。在后一种情况下，write()调用被延迟，直到指示底层网络准备就绪的事件到达。

生产级应用服务器必须处理带宽管理，特别是网关类型的服务器，它们位于具有不同吞吐量的网络(如数据中心网络和移动网络)之间。随着出站缓冲区的增长，节流为入站事件处理提供了一个有用的反馈循环。

可以通过一种非常简单的方法看到令人满意的结果，例如当出站缓冲区达到一定大小时,入站读取事件的暂时“暂停”，以及当出站缓冲区大小在另一个阈值下减少时的“恢复”操作。

单线程多线程

与更高的代码复杂度相关的一个常见性能损失来自于应用服务器的多线程特性。线程必须为同步访问各种重要资源(如内存、文件描述符或事件日志)的锁和互斥锁而相互竞争。这样的同步对于适当的应用程序功能非常重要，但是它需要大量可用的资源，比如CPU容量。

多线程应用服务器不会以线性方式伸缩。您最终会到达这样一个点:添加更多的CPU内核不会对性能产生任何积极的影响。

由于这些潜在的问题，我更喜欢单线程设计。 它们或多或少地线性扩展，不受主要类型的竞争条件的影响 - 这是多线程网络I / O应用程序的常见缺陷。 另一个好处是代码简单明了。

由于专用进程(也称为worker)，单线程设计能够彼此并行工作。每个工作进程都是单线程的，并处理应用程序事务的一部分。

工作进程可以很容易地在一段时间后重新启动，例如，如果需要减少内存碎片的话。

内存和零拷贝

内存也很重要。预先分配的用于存储事务数据的固定帧内存池往往工作得很好。

应用程序服务器也应尽可能避免数据复制操作。 因此，部分协议实现可以构建为标题/尾部添加/删除。 然后，数据在整个事务期间保持在相同的内存位置。 向量化I / O（分散/聚集I / O）是另一种有效的方法，可帮助避免不必要的复制操作。 [5]

该规则的一个显著例外是数据的加密和解密，这本质上是一个复制操作。

日志

令人惊讶的是，在高性能服务器中，日志记录操作成本很高。 日志提供了对服务器操作的必要见解，因此不应完全牺牲它们。 但是，一些天真的程序员倾向于意外地引入瓶颈，例如与日志文件写入相关的附加I / O操作。

更可行的方法是在活动应用程序事务期间登录到内存中，并以更大的块刷新内存。这可能会在日志记录中引入延迟，但是您可以通过选择一个良好的刷新间隔来缓解这个问题。200ms通常是一个很好的起点。

此外，现代日志处理工具处理具有非线性时间戳的日志流，没有任何问题。

你对未来有什么期待

我不会在这里停下来。 该领域有许多主题需要探索。 我开始使用协议链，允许在不影响性能的情况下轻松，标准化地实现各种分层协议，例如TCP，TLS和HTTP链。 [9]

您可以通过跳过Linux网络堆栈并直接在用户空间中处理原始数据包来避免这种麻烦。 这个领域有项目，例如DPDK。 [10]这种方法必须与TCP / IP堆栈的用户空间实现相结合，这可能会导致很多问题。 但是，这是值得的，因为产生的吞吐量性能非常出色。

下一个逻辑步骤是研究基于这种体系结构的TLS实现，因为作为OpenSSL一部分的标准库(如SSL)并没有为这种数据流量提供最佳工具。
