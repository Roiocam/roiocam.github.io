记一次为 Spring Boot 提交Bug

Posted by Andy Chen on 2019-08-04

缘起

公司的API接口项目中使用了 Spring Boot 构建，其中因为包含上传文件接口，故使用了 JBoss 的 Undertow (由 Spring Boot 支持). 并且配置了上传文件的临时目录: spring.servlet.multipart.location=

由于项目的其他同事都是使用的Windows系统，故在配置文件上，此属性为Windows的绝对路径，但是我是Manjaro和Windows都用，所以启动项目时，发现了这样一个异常。

Exception Stacktrace:

通过观察异常日志，然后发现异常在这里抛出io.undertow.servlet.core.ManagedServlet.setupMultipart(ManagedServlet.java:107).通过Debug断电发现，

setupMultipart方法的这一段代码发生了异常：

通过debug分析得， locFile.isAbsolute() ==> 这个方法因OS不同导致的识别结果不同，在Windows下/usr/local….这样的路径无法被识别为绝对路径，在Linux下，C://Windows….这样的路径无法被识别为绝对路径，所以执行了else部分的代码，在else中的所有对象中，DeploymentInfo的tempDir没有设置默认值,导致NPE的发生，所以我首先判断出这是Undertow的锅，因为异常最终在ManagedServlet抛出，所以我去https://issues.jboss.org 提交了Issue，并提供了如何复现以及异常日志。很快我得到了解决方案，抛出异常的代码被加上了是否为null的判断。

深入探讨

我看了一下这个解决方案的PR，随后以软件工程的思想分析了一下，我重新复现场景并Debug，发现在DeploymentInfo对象是由Spring Boot创建的，并且在DeploymentInfo的构造方法中，tempDir不是作为必要属性。故猜想是Spring Boot 没有为这个属性创建默认值，溯原后发现是UndertowServletWebServerFactory的createDeploymentManager创建并调用，导致了NPE的产生，故FORK了Spring Boot项目，在createDeploymentManager方法中为创建的DeploymentInfo对象设置系统的临时目录。并且在Github上提交了带复现以及描述的PR。
