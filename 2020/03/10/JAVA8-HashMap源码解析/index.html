JAVA8 HashMap源码解析

Posted by Andy Chen on 2020-03-10

一. JAVA7的HashMap问题

  1. 扩容时的插入顺序问题,导致多线程下会产生死锁.
  2. 防止哈希碰撞的机制简陋

为了解决这两个问题,JAVA8做了大量改进,和一些新增.

前置知识

  1. 实现变为 : 数组 + 链表 / 红黑树
  2. 扩容时插入顺序改进
  3. 函数方法(forEach,compute)
  4. Map新API
  5. 防止哈希碰撞的机制变更.

二. JAVA8 HashMap源码解析

类注解

HashMap类注解的信息比较长,本来本文写了翻译的.但是考虑到篇幅于是只进行概括.

  1. 普通情况下为桶结构的哈希表,当链表过长时将链表树化为红黑树
  2. 链表长度为8时树化(TREEIFY_THRESHOLD)，树节点小于6时(UNTREEIFY_THRESHOLD)退化。

TREEIFY_THRESHOLD和UNTREEIFY_THRESHOLD是JAVA8的HashMap新增的两个常量,DEFAULT_INITIAL_CAPACITY必须为2的n次幂我们已经知道,是基于indexFor方法和位运算的一些性质决定,而TREEIFY_THRESHOLD为什么如此呢?我们也要搞清楚.

TREEIFY_THRESHOLD 和 UNTREEIFY_THRESHOLD

首先我们先看类注解中有没有提示.

* Because TreeNodes are about twice the size of regular nodes, we
* use them only when bins contain enough nodes to warrant use
* (see TREEIFY_THRESHOLD). And when they become too small (due to
* removal or resizing) they are converted back to plain bins.  In
* usages with well-distributed user hashCodes, tree bins are
* rarely used.  Ideally, under random hashCodes, the frequency of
* nodes in bins follows a Poisson distribution
* (http://en.wikipedia.org/wiki/Poisson_distribution) with a
* parameter of about 0.5 on average for the default resizing
* threshold of 0.75, although with a large variance because of
* resizing granularity. Ignoring variance, the expected
* occurrences of list size k are (exp(-0.5) * pow(0.5, k) /
* factorial(k)). The first values are:
*
* 0:    0.60653066
* 1:    0.30326533
* 2:    0.07581633
* 3:    0.01263606
* 4:    0.00157952
* 5:    0.00015795
* 6:    0.00001316
* 7:    0.00000094
* 8:    0.00000006
* more: less than 1 in ten million

注解中写到（理想情况下，在随机哈希码和默认大小调整阈值为0.75的情况下,存储桶中元素个数出现的频率遵循泊松分布）并且注解给了Lambda平均约为0.5的估算值.(Lambda为单位时间内随机事件的平均发生率,可用统计学的最大似然估计来计算得出).

并且注解指出,由于大小调整粒度(在HashMap中为容量*负载因子,大于此值则重新调整存储桶的个数,故k值因resize()变化而不稳定,故k值大小相差较大)导致方差较大。故忽略方差,将Lambda代入泊松分布公式

((exp(-0.5) * pow(0.5, k) / factorial(k)))

可得,链表大小有如上注解概率。

之所以树化阈值选定8,一方面是因为红黑树在节点数量小时效率不一定高(可看HashMap中的TreeNode,占用了4个引用,1个布尔的内存空间,而链表节点则更低.)。另一方面是根据泊松分布计算出来,随机哈希码下,链表大小出现的概率。猜想是由这两方面和一些考量之后得出的结果。

之所以退化阈值选定6,一方面是此阈值必须低于8,选定7的话,极端情况下(如一张链表的大小频繁的从8到7，7到8)导致频繁树化，影响性能

树化和退化的判断

带着上面的理论知识我们来看HashMap是如何判断树化和退化的。

判断链表树化

#树化
//遍历链表,
for (int binCount = 0; ; ++binCount) {
    //已无下一个元素,直接在链表后面新增
    if ((e = p.next) == null) {
        p.next = newNode(hash, key, value, null);
    //若链表长度超过阈值,树化.
    if (binCount >= TREEIFY_THRESHOLD - 1) // -1 for 1st
        treeifyBin(tab, hash);
        break;
    }
    //找到相同元素,直接跳出循环,执行循环外的替换方法
    if (e.hash == hash && ((k = e.key) == key || (key != null && key.equals(k))))
        break;
    p = e;
}
final void treeifyBin(Node<K,V>[] tab, int hash) {
    int n, index; Node<K,V> e;
    //当哈希表容量未达到一定大小时，考虑调整大小而非树化链表
    if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY)
        resize();
    else if ((e = tab[index = (n - 1) & hash]) != null) {
        TreeNode<K,V> hd = null, tl = null;
        do {
            TreeNode<K,V> p = replacementTreeNode(e, null);
            if (tl == null)
                hd = p;
            else {
                p.prev = tl;
                tl.next = p;
            }
            tl = p;
        } while ((e = e.next) != null);
        if ((tab[index] = hd) != null)
            hd.treeify(tab);
    }
}

由代码得知,HashMap内部主要通过UNTREEIFY_THRESHOLD和TREEIFY_THRESHOLD两个属性来判断为什么退化代码中有一个低位链表概念.当哈希表容量未达到一定数量时，考虑调整大小而非树化链表。

resize()方法中的高低位链表解析

final Node<K,V>[] resize() {
    Node<K,V>[] oldTab = table;
    int oldCap = (oldTab == null) ? 0 : oldTab.length;
    int oldThr = threshold;
    int newCap, newThr = 0;
    if (oldCap > 0) {
        //当前容量大于最大容量.直接返回
        if (oldCap >= MAXIMUM_CAPACITY) {
            threshold = Integer.MAX_VALUE;
            return oldTab;
        }
        else if ((newCap = oldCap << 1) < MAXIMUM_CAPACITY &&
                 oldCap >= DEFAULT_INITIAL_CAPACITY)
            newThr = oldThr << 1; // double threshold
    }
    //空HashMap,使用阈值初始化容量
    else if (oldThr > 0) // initial capacity was placed in threshold
        newCap = oldThr;
    //空HashMap,使用默认阈值初始化容量
    else {               // zero initial threshold signifies using defaults
        newCap = DEFAULT_INITIAL_CAPACITY;
        newThr = (int)(DEFAULT_LOAD_FACTOR * DEFAULT_INITIAL_CAPACITY);
    }
    if (newThr == 0) {
        float ft = (float)newCap * loadFactor;
        newThr = (newCap < MAXIMUM_CAPACITY && ft < (float)MAXIMUM_CAPACITY ?
                  (int)ft : Integer.MAX_VALUE);
    }
    threshold = newThr;
    @SuppressWarnings({"rawtypes","unchecked"})
    Node<K,V>[] newTab = (Node<K,V>[])new Node[newCap];
    table = newTab;
    if (oldTab != null) {
        //遍历旧的桶数组
        for (int j = 0; j < oldCap; ++j) {
            Node<K,V> e;
            if ((e = oldTab[j]) != null) {
                oldTab[j] = null;
                //当前桶为空,则新增桶
                if (e.next == null)
                    newTab[e.hash & (newCap - 1)] = e;
                //当前桶为红黑树,将红黑树分离
                else if (e instanceof TreeNode)
                    ((TreeNode<K,V>)e).split(this, newTab, j, oldCap);
                //使用高低位链表来分离增大一倍后元素的去向。
                else { // preserve order
                    Node<K,V> loHead = null, loTail = null;
                    Node<K,V> hiHead = null, hiTail = null;
                    Node<K,V> next;
                    do {
                        next = e.next;
                        if ((e.hash & oldCap) == 0) {
                            if (loTail == null)
                                loHead = e;
                            else
                                loTail.next = e;
                            loTail = e;
                        }
                        else {
                            if (hiTail == null)
                                hiHead = e;
                            else
                                hiTail.next = e;
                            hiTail = e;
                        }
                    } while ((e = next) != null);
                    if (loTail != null) {
                        loTail.next = null;
                        newTab[j] = loHead;
                    }
                    if (hiTail != null) {
                        hiTail.next = null;
                        newTab[j + oldCap] = hiHead;
                    }
                }
            }
        }
    }
    return newTab;
}

JAVA8 HashMap源码解析

1. 改进的防止哈希碰撞机制的hash()方法

JAVA8的hash方法比较简单,没有那么多的骚操作.

根据注解:

计算key.hashCode() 并且将hashCode的高位异或到低位去。(因为表使用的是2的幂次方的掩码,仅在当前掩码以上位变化的HashCode将一直碰撞).所以我们应用了一个将高位比特变换到低位比特的转换。在位传递(Bit Spread)时有（速度，实用和质量）的trade off. 因为许多常见的散列集合已经合理的分布了，并且我们使用树来处理集合中大量的哈希碰撞，所以我们只是用最便宜的方式异或一些已经移动的Bits 来降低系统损耗,并且将高位加入运算中,否则高位永远不参与运算。

注解有些冗长. 简单来说就是将 (哈希值) 去和 (哈希值无符号右移) 异或.

源码:

static final int hash(Object key){
    int h;
    //使用哈希值 异或(^) 哈希值的高位. 
    return (key == null) ? 0 : (h == key.hashCode()) ^ ( h >>> 16);
}
#举例 如key的hashCode = 1101 0101
则方法实际为 (1101 0101) ^ (0000 1101)

2. 插入方法

此方法代码有点长,约40行.(putVal()方法,之所以使用putVal()来做真正的插入操作是因为,Map接口的put方法签名无法满足HashMap中的需求.)

由注解我们得知,JDK1.8之后,HashMap的实现由(数组+链表)变成了(数组+链表/红黑树).其机制是当链表达到一定长度时树化,而红黑树节点少于一定数量时退化到链表。

源代码:

final V putVal(int hash, K key, V value, boolean onlyIfAbsent, boolean evict) {
    Node<K,V>[] tab; Node<K,V> p; int n, i;
    //空表，则初始化
    if ((tab = table) == null || (n = tab.length) == 0){
        n = (tab = resize()).length;
    }
    //若找到了该桶，但是第一个元素，则直接插入新节点
    if ((p = tab[i = (n - 1) & hash]) == null){
        tab[i] = newNode(hash, key, value, null);
    }else {
        Node<K,V> e; K k;
        //若当前节点是否就是需要填入的key,是则覆盖.
        if (p.hash == hash && ((k = p.key) == key || (key != null && key.equals(k))))
            e = p;
        //若当前节点为树节点,则执行树节点的插入操作    
        else if (p instanceof TreeNode)
            e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value);
        //否则为链表
        else {
            //在链表中找
            for (int binCount = 0; ; ++binCount) {
                //在map中没找到该key
                if ((e = p.next) == null) {
                    //插入节点
                    p.next = newNode(hash, key, value, null);
                    //若当前链表节点个数超过变树阈值.则将
                    if (binCount >= TREEIFY_THRESHOLD - 1) // -1 for 1st
                        treeifyBin(tab, hash);
                    break;
                }
                //若已存在,则覆盖.
                if (e.hash == hash &&
                    ((k = e.key) == key || (key != null && key.equals(k))))
                    break;
                p = e;
            }
        }
        if (e != null) { // existing mapping for key
            V oldValue = e.value;
            if (!onlyIfAbsent || oldValue == null)
                e.value = value;
            afterNodeAccess(e);
            return oldValue;
        }
    }
    ++modCount;
    if (++size > threshold)
        resize();
    afterNodeInsertion(evict);
    return null;
}

3. resize(); 扩容

源代码:

final Node<K,V>[] resize() {
    Node<K,V>[] oldTab = table;
    int oldCap = (oldTab == null) ? 0 : oldTab.length;
    int oldThr = threshold;
    int newCap, newThr = 0;
    if (oldCap > 0) {
        if (oldCap >= MAXIMUM_CAPACITY) {
            threshold = Integer.MAX_VALUE;
            return oldTab;
        }
        else if ((newCap = oldCap << 1) < MAXIMUM_CAPACITY &&
                 oldCap >= DEFAULT_INITIAL_CAPACITY)
            newThr = oldThr << 1; // double threshold
    }
    else if (oldThr > 0) // initial capacity was placed in threshold
        newCap = oldThr;
    else {               // zero initial threshold signifies using defaults
        newCap = DEFAULT_INITIAL_CAPACITY;
        newThr = (int)(DEFAULT_LOAD_FACTOR * DEFAULT_INITIAL_CAPACITY);
    }
    if (newThr == 0) {
        float ft = (float)newCap * loadFactor;
        newThr = (newCap < MAXIMUM_CAPACITY && ft < (float)MAXIMUM_CAPACITY ?
                  (int)ft : Integer.MAX_VALUE);
    }
    threshold = newThr;
    @SuppressWarnings({"rawtypes","unchecked"})
    Node<K,V>[] newTab = (Node<K,V>[])new Node[newCap];
    table = newTab;
    if (oldTab != null) {
        for (int j = 0; j < oldCap; ++j) {
            Node<K,V> e;
            if ((e = oldTab[j]) != null) {
                oldTab[j] = null;
                if (e.next == null)
                    newTab[e.hash & (newCap - 1)] = e;
                else if (e instanceof TreeNode)
                    ((TreeNode<K,V>)e).split(this, newTab, j, oldCap);
                //保持链表顺序,但这样只是让并发时出现问题的概率变低，并非并发安全。
                else {
                    Node<K,V> loHead = null, loTail = null;
                    Node<K,V> hiHead = null, hiTail = null;
                    Node<K,V> next;
                    //扩容(2的n次幂进一次幂)时,比特位数+1,则源hashCode和新的hashCode,在原位数上不变,在新增的位数上,不是1就是0,在这里将0变成loHead,1的变成hiHead，分为高低位链表.这种操作保持了源链表的顺序
                    do {
                        next = e.next;
                        //最高那一位是否为0(决定进高位链表还是低位链表)
                        if ((e.hash & oldCap) == 0) {
                            if (loTail == null)
                                loHead = e;
                            else
                                loTail.next = e;
                            loTail = e;
                        }
                        else {
                            if (hiTail == null)
                                hiHead = e;
                            else
                                hiTail.next = e;
                            hiTail = e;
                        }
                    } while ((e = next) != null);
                    //将低位链表和高位链表分别赋值给新的数组(新的哈希桶)里面去
                    if (loTail != null) {
                        loTail.next = null;
                        newTab[j] = loHead;
                    }
                    if (hiTail != null) {
                        hiTail.next = null;
                        newTab[j + oldCap] = hiHead;
                    }
                }
            }
        }
    }
    return newTab;
}
